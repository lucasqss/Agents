# ADR 001: Taxonomia de Pastas para Microsserviços

**Status:** Aceito
**Data:** 25 de maio de 2026
**Autor:** Lucas de Queiroz

> Este ADR é genérico por design: o pacote raiz aparece como `org.acme` em todos os exemplos. Cada adoção deve substituir `org.acme` pelo prefixo da organização (ex.: `com.empresa.produto`).

## 1. Contexto e Problema

Com o desenvolvimento contínuo de soluções baseadas em Java e Quarkus, a falta de uma taxonomia de pastas padronizada gera atrito. Desenvolvedores enfrentam dúvidas recorrentes sobre onde alocar classes (DTOs, Mappers, Entidades JPA vs. Entidades de Domínio), resultando em arquiteturas mistas que dificultam a manutenção, o onboarding de novos membros e o isolamento de testes.

Precisamos de uma estrutura pragmática que suporte a separação de responsabilidades (Clean Architecture / Ports and Adapters) nas aplicações com regras de negócio relevantes, sem impor engenharia excessiva em componentes focados apenas em I/O e repasse de dados.

**Nota sobre nomenclatura:** este ADR adota deliberadamente os termos `passthrough`, `core` e `acl`. A escolha de `acl` (Anti-Corruption Layer, no sentido canônico do DDD) para o perfil de tradução bidirecional substitui o termo "Adapter" original, que conflita com a terminologia Hexagonal usada cotidianamente (todo `infra/client/` é, tecnicamente, um adapter de saída).

## 2. Decisão

Adotaremos uma taxonomia baseada em **três perfis arquiteturais**. Cada componente deve seguir estritamente a estrutura correspondente ao seu papel.

A descrição de cada pacote vive em `package-info.java` (Javadoc) — não em arquivos `.info` arbitrários. A IDE renderiza, o compilador valida, e ferramentas de doc consomem.

### Perfil A — `passthrough` (I/O síncrono, sem estado, sem regras)

Otimizado para componentes que apenas chamam APIs externas e devolvem resposta traduzida. Sem domínio, sem persistência.

```text
src/main/java/org/acme/passthrough
├── entrypoint/
│   ├── rest/           # Controladores JAX-RS expostos externamente.
│   └── exception/      # ExceptionMappers (rede → HTTP).
├── dto/
│   ├── request/        # Entrada com Bean Validation.
│   └── response/       # Resposta entregue ao consumidor.
├── service/
│   ├── mapper/         # Tradução DTO externo ↔ modelo interno.
│   └── orchestrator/   # Coordena: controller → cliente → mapper.
└── infra/
    ├── client/         # @RegisterRestClient / stubs gRPC.
    └── config/         # Produtores CDI, Jackson, config de ambiente.
```

### Perfil B — `core` (domínio rico; "camada negocial")

Para serviços com invariantes de negócio próprias, estado, persistência relevante e agregados.

```text
src/main/java/org/acme/core
├── entrypoint/
│   ├── rest/           # Controladores JAX-RS síncronos.
│   ├── kafka/          # @Incoming consumers.
│   ├── scheduler/      # @Scheduled (opcional por serviço).
│   └── dto/            # Contratos externos isolados do domínio.
├── application/
│   ├── service/        # Orquestração, @Transactional.
│   └── exception/      # Exceções de fluxo (ex: RecursoNaoEncontrado).
├── domain/
│   ├── model/          # @Entity JPA + Value Objects (ver Seção 5).
│   ├── event/          # Domain Events (records imutáveis, nome no pretérito).
│   ├── port/           # Interfaces que o domínio precisa do externo.
│   └── exception/      # Violações de regra de negócio.
└── infra/
    ├── persistence/    # PanacheRepository (quando Repository pattern; entidades NÃO ficam aqui).
    ├── facade/         # Implementações de ports do domínio para integrações pesadas.
    ├── client/         # HTTP / mensageria de saída.
    └── config/         # Produtores CDI e config.
```

### Perfil C — `acl` (Anti-Corruption Layer; tradução bidirecional)

Recebe payloads externos (webhook/Kafka), traduz para o vocabulário interno via regras puras e despacha para o `core`. Sem estado próprio, sem persistência.

```text
src/main/java/org/acme/acl
├── entrypoint/
│   ├── rest/           # Webhooks recebidos.
│   ├── kafka/          # Consumers de tópico externo.
│   └── dto/            # Schema bruto do parceiro.
├── application/
│   └── service/        # Orquestra: receber → traduzir → despachar.
├── domain/
│   ├── model/          # Enums/VOs do vocabulário interno.
│   └── policy/         # Regras puras de tradução (sem framework).
└── infra/
    ├── mapper/         # Usa policy para converter DTO externo → modelo interno.
    ├── client/         # Saída para o core (REST/Kafka).
    └── config/         # Produtores CDI e config.
```

## 3. Matriz de Decisão por Perfil

| Sintoma | Perfil |
| --- | --- |
| Tem invariantes de negócio próprias, estado, persistência relevante, agregados | `core` |
| Apenas chama API parceira e devolve resposta traduzida (síncrono, sem estado) | `passthrough` |
| Recebe evento/webhook externo, traduz para vocabulário interno e despacha para outro serviço | `acl` |

**Regra de bolso:** na dúvida entre `passthrough` e `core`, escolher `passthrough` até surgir a primeira invariante de negócio. Promover depois é mais barato que extrair lógica de um Core que não precisava existir.

## 4. Fitness Functions (ArchUnit)

A taxonomia ganha aderência executável via [ArchUnit](https://www.archunit.org/). As regras são testes JUnit que falham o build se a estrutura derivar do ADR. Rodam no `test` do Maven na pipeline `pr-*` (commit stage ≤ 10 min).

### Dependência (uma vez por projeto, em `pom.xml`)

```xml
<dependency>
    <groupId>com.tngtech.archunit</groupId>
    <artifactId>archunit-junit5</artifactId>
    <version>1.3.0</version>
    <scope>test</scope>
</dependency>
```

### Estrutura de testes

```text
src/test/java/org/acme/<perfil>/
└── architecture/
    ├── CommonArchitectureTest.java
    └── <Perfil>ArchitectureTest.java
```

### Regras comuns (`CommonArchitectureTest`)

- `domain/..` **não depende de** `jakarta.ws.rs..`, `org.eclipse.microprofile.rest.client..`, nem `io.quarkus..` de transporte.
- `entrypoint/..` **não depende de** `jakarta.persistence..` nem `io.quarkus.hibernate..`.
- Slices de primeiro nível (`entrypoint`, `application`, `domain`, `infra`) formam grafo **acíclico** (`slices().should().beFreeOfCycles()`).
- Classes anotadas com `@Path` residem **apenas em** `..entrypoint.rest..`.
- Classes anotadas com `@Incoming`/`@Outgoing` residem **apenas em** `..entrypoint.kafka..` ou `..infra.client..`.

### Perfil `core` (`CoreArchitectureTest`)

- Classes `@Entity` residem **apenas em** `..domain.model..`.
- Subclasses de `PanacheRepositoryBase` residem **apenas em** `..infra.persistence..`.
- Métodos públicos `@Transactional` residem **apenas em** `..application.service..`.
- Classes em `..domain.event..` são `record` (`should().beRecords()`).
- `..domain..` **não depende de** `..application..` nem `..infra..`.

### Perfil `passthrough` (`PassthroughArchitectureTest`)

- Não existem classes em `..domain..` (`noClasses().should().resideInAPackage("..domain..")`).
- Classes `@RegisterRestClient` residem **apenas em** `..infra.client..`.
- Não existem `@Entity` nem subclasses de `PanacheRepositoryBase`.

### Perfil `acl` (`AclArchitectureTest`)

- `..domain.policy..` **não depende de** `jakarta..`, `io.quarkus..`, `org.eclipse.microprofile..` (funções puras).
- Não existem classes anotadas com `@Entity` nem `@Transactional`.
- Classes em `..infra.mapper..` **não dependem de** `..entrypoint..`.

Falha em qualquer regra = build vermelho = não merge.

## 5. Estratégia de Persistência: Modelo Único

`@Entity` JPA **é** o objeto de domínio. Vive em `domain/model/` com comportamento rico (métodos de negócio, invariantes, factory methods). Não há dois modelos paralelos com mapper entre eles.

**Trade-off aceito:** acoplamento controlado do domínio a `jakarta.persistence`. Em troca, eliminamos a sobrecarga de duas hierarquias e o boilerplate de mapeamento — Quarkus + Hibernate Panache foram desenhados para esse modelo, e a regra do ArchUnit garante que o domínio não vaza para outros pacotes de framework.

**`infra/persistence/`** guarda apenas `PanacheRepository` quando o serviço opta por Repository pattern em vez de Active Record (Panache permite ambos).

**Quando reavaliar:** se um agregado crescer ao ponto de exigir invariantes não-representáveis em JPA (ex.: estado interno que JPA não consegue carregar coerentemente), abrir ADR de revisão para esse contexto específico — não para o time todo.

## 6. Consequências

**Ganhos:**
- Onboarding mais rápido: três perfis cobrem o catálogo de serviços.
- ArchUnit converte o ADR em artefato executável — sem ele, a taxonomia eroderia em 2 sprints.
- Modelo único de persistência reduz boilerplate sem perder isolamento (garantido pelas regras).
- `entrypoint/` unifica todas as formas de entrada (REST, Kafka, scheduler), cobrindo o caso "worker/batch" sem criar um quarto perfil.

**Abrimos mão de:**
- Pureza Clean Architecture estrita (JPA no domínio).
- Flexibilidade de organizar pastas livremente por contexto — a taxonomia é normativa.

**Reavaliar quando:**
- Um serviço crescer ao ponto de exigir invariantes não-representáveis em JPA (Seção 5).
- Surgir um perfil arquitetural recorrente que não caiba em nenhum dos três (ex.: BFF dedicado, gateway de agregação).

## 7. Follow-ups (não cobertos por este ADR)

- Estrutura de `src/test/java` para testes de negócio (unitários, integração, Pact, Testcontainers).
- Estratégia de versionamento de API (OpenAPI, `Deprecation`, `Sunset`).
- Política de feature flags e padrões de migração (Strangler Fig, Branch by Abstraction).
